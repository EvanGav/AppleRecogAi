## Architecture

### Why an anchor-free detector?

Standard SSD requires carefully tuned anchor boxes (predefined sizes and aspect ratios per grid cell). An anchor-free approach assigns each cell a single prediction — simpler to implement, easier to debug, and sufficient for a single-class detector where apple sizes are relatively uniform within the dataset.

### Why MobileNetV2 as backbone?

MobileNetV2 is pre-trained on ImageNet and already has strong feature representations for round, textured objects. Freezing it and only training the small detection head reduces trainable parameters dramatically — critical when the dataset has only 670 images. Training a backbone from scratch on 670 images would overfit within 1–2 epochs. (strategy given by Claude)

---

### MineApple problems

MinnEapple masks encode apple instances as **pixel values** (1, 2, 3… one integer per apple instance), not binary black/white images. The binarisation threshold must be `> 0`, not the typical `> 127`. Using 127 as the threshold silently zeroes every apple pixel, producing a dataset with zero labels — the single most impactful bug discovered in this project. (I asked Claude to explain why that worked after finding it randomly)

```python
# WRONG — all apple pixels (value 1, 2, 3) become black
cv2.threshold(mask, 127, 255, cv2.THRESH_BINARY)

# CORRECT — any non-zero pixel is an apple instance
cv2.threshold(mask, 0, 255, cv2.THRESH_BINARY)
```

## Training

### Strategy

The MobileNetV2 backbone stays **fully frozen** throughout training. With only ~530 training images (80/20 split), any gradient flowing into the backbone's 2.2M parameters causes overfitting within 1–2 epochs, the model memorises training image textures rather than learning what apples look like in general.

**Phase 1** (epochs 1–15): frozen backbone, head-only at LR = 1e-3

**Phase 2** (epochs 16–20): frozen backbone, LR = 2e-4 for fine refinement

### Callbacks

| Callback | Configuration |
|---|---|
| `ModelCheckpoint` | Saves best weights by `val_loss`, weights-only format |
| `EarlyStopping` | Patience 7 epochs, restores best weights |
| `ReduceLROnPlateau` | Halves LR after 3 stagnant epochs, min LR = 1e-7 |
| `TensorBoard` | Loss curves logged per epoch to `logs/` |

## Limitations

- **Small dataset.** 670 images is small for object detection. Most production detectors use tens of thousands of images. The frozen backbone mitigates but does not eliminate this constraint.

- **Single prediction per cell.** If two apples share a grid cell, only one is detected. Anchor-based detectors (SSD, Faster R-CNN) or transformer detectors (DETR) handle this correctly.

- **Size problem** MinneApple images show small distant apples, the model has never seen close up photos of apples. The `BOX_SCALE` post-processing factor partially compensates but cannot fully correct this.

- **mAP metric.** The dashboard mAP of 0.0% reflects that predicted boxes do not overlap ground truth at IoU ≥ 0.5 — the standard COCO threshold. The model detects apple regions correctly, but box precision is not high enough to pass this strict criterion. At IoU 0.3 the mAP would be significantly higher.

- **CPU training.** Without GPU, experiments are limited to ~20 epochs at 2 minutes each. GPU training would allow more.

## Lessons learned


**Mask threshold must be 0, not 127.** MinneApple instance masks use pixel values 1, 2, 3… Thresholding at 127 silently zeroes every apple pixel. The model trained for 10 epochs on empty targets with zero detections.

**Box loss must be masked to positive cells only.** Without the positive mask, background cells dominate the gradient. The model learns "predict zero everywhere"

**sigmoid × 0.5 is more stable than exp() for size prediction.** `exp()` caused `val_box_loss: 162,000` on the first epoch. Log-space targets also failed. Predictions via `sigmoid × 0.5` caps outputs at [0, 0.5] with zero risk of explosion.

**Focal loss needs careful weight initialisation.** Default random weights produce objectness logits around −10 : `sigmoid(−10) ≈ 0` gives `BCE ≈ 0` for background cells, no signal at all. Initialising with `kernel_std=0.01` and `bias=0` brings initial logits near 0, giving `sigmoid(0) = 0.5` 

**val_loss: 0.0000 is not good.** Early runs showed instant convergence to zero loss. This was the opposite of learning — the model predicted zeros everywhere, the masked loss had no positive cells, and the result was perfectly useless weights. There will always be somme loss

**The Keras loss dict API cannot mask by cell.** Using `model.compile(loss={"box_output": HuberLoss()})` applies the loss uniformly to all cells. Correctly masking the box loss to positives only requires a custom `train_step` which is how YOLO-family detectors actually implement their loss.
